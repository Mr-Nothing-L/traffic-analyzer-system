# 自动化高速交通事件判别系统 — Agent 工作流设计文档

**版本**: V2.0  
**适用范围**: 高速公路监控视角视频（双向/单向）  
**修订说明**: 基于应急车道倒车事件教训修订，新增自适应采样、精采样重判验证、位移方向优先等核心机制。

---

## 一、系统概述与输入输出

### 输入

一段高速公路监控视角视频。视频中通常为双向行驶的高速车道，也可能因 FOV（视场角）限制仅显示单向行驶画面。视频可能包含不同天气条件、光照条件和交通密度。

### 输出

结构化的事件判别报告，包含以下核心内容：

- 道路结构分析（主路数量、坐标范围）
- 每条主路的正常行驶方向（含验证）
- 全部可追踪车辆的跨帧位移轨迹
- 异常事件判定结果（逆行/倒车/违停/应急车道占用）
- 多层验证机制的执行结果
- 可视化证据帧与处置建议

---

## 二、Step 1：视频拆帧（自适应采样）

视频拆帧是整个工作流的基础步骤。为适应不同速度车辆的检测需求，采用**自适应双轮采样策略**。

### 2.1 第一轮粗采样

1. 提取视频基本信息：帧率 FPS、分辨率、总帧数、时长。
2. 按 **每秒 1 帧** 进行均匀采样，将所有帧保存至本地工作目录，命名格式为 `frame_{秒数}.jpg`。
3. 粗采样用于快速建立视频时间轴概览，识别明显的运动目标和静止区域。

### 2.2 停留车辆区域识别

遍历粗采样帧，执行以下检测逻辑：

1. 对相邻帧进行帧差法或背景减除，提取运动区域。
2. 识别画面中存在车辆轮廓但跨帧位移 **< 5 像素** 的候选区域。
3. 若候选区域持续存在超过 **2 秒**，标记为"潜在停留区域"，记录其时区范围。

### 2.3 第二轮精采样

对 2.2 中标记的"潜在停留区域"所在时区，提升采样密度至 **每秒 4 帧**（即每 0.25 秒一帧）。

- 精采样覆盖范围应包含停留起始前 2 秒至结束后 2 秒，以确保捕获完整的位移趋势。
- 精采样帧命名格式为 `frame_{秒数}_{子帧索引}.jpg`，子帧索引范围 0-3。

> **核心教训**：粗采样（1 FPS）无法可靠捕捉慢速倒车行为。对停留车辆必须提升至 4 FPS 精采样，才能通过逐帧位移检测出倒车趋势。应急车道倒车的白色小车即因此教训而被遗漏。

## 三、Step 2：主路数量判定

根据画面中的道路物理结构，将监控视角划分为若干独立的主路区域。

### 3.1 道路结构分析

选取无车辆严重遮挡的清晰帧，执行以下检测：

- **中央隔离带检测**：检测绿化带、防眩板、水泥墩等物理隔离结构的位置与宽度。
- **车道标线检测**：利用霍夫直线变换检测车道分隔线的分布区域与走向。
- **路面区域分割**：根据隔离带和标线将画面划分为独立的主路候选区域。

### 3.2 主路区域划分

记录每条主路的像素坐标范围，格式如下：

| 主路编号 | 画面坐标范围 (x, y) |
|---------|-------------------|
| 主路1 | 左侧道路区域，例如 x=0~950, y=100~1080 |
| 主路2 | 右侧对向道路区域，例如 x=1200~1920, y=100~500 |
| 中央隔离带 | 物理分隔区域，例如 x=950~1200（非行车区域） |

### 3.3 FOV 不足场景处理

若画面中仅显示单侧道路（无对向车道可见），则主路数量记为 **1 条**。此时所有分析仅针对单主路执行，Step 3 的互补验证逻辑自动跳过。

---

## 四、Step 3：主路行驶方向判定

根据各主路内大量车辆的移动情况，判定该主路的正常行驶方向。此步骤为后续逆行/倒车判定提供**基准方向**。

### 4.1 车辆位移采样

在每个主路区域内，选取至少 **3 辆** 处于行驶状态的车辆作为方向采样样本。对每辆样本车辆：

- 追踪其在连续多帧中的中心坐标 (cx, cy)。
- 计算跨帧位移向量 (dx, dy)。
- 优先选取处于不同车道、不同时间窗口的车辆，以避免局部异常干扰。

### 4.2 方向统计计算

对每个主路，汇总所有样本车辆的 Y 方向位移：

- 若 **avg_dy > 0**（多数车辆 Y 坐标增加）→ 该主路行驶方向定义为 **Y 增加方向**（由远及近 / 朝向摄像头）
- 若 **avg_dy < 0**（多数车辆 Y 坐标减小）→ 该主路行驶方向定义为 **Y 减小方向**（由近及远 / 远离摄像头）

### 4.3 互补验证

高速公路为双向对向行驶结构，两侧主路的行驶方向理论上必须相反：

- 若主路1判定为 Y 增加方向，则主路2必须为 Y 减小方向。
- 若主路1判定为 Y 减小方向，则主路2必须为 Y 增加方向。
- 若验证不通过（两侧同向），返回 Step 3 重新采样更多车辆，或检查主路划分是否正确。
- 若某一主路在观测时段内无任何行驶车辆，按其反侧主路方向进行逻辑推断，并在报告中标注为"推断方向"，置信度标记为"中"。

| 主路 | 坐标范围 | 平均位移 dy | 判定方向 | 验证状态 |
|------|---------|------------|---------|---------|
| 主路1 | x=0~950 | +215.6 像素 | Y增加（向南） | 通过 |
| 主路2 | x=1200~1920 | -43.3 像素 | Y减小（向北） | 通过 |
| 互补验证 | - | - | 两侧方向相反 | 通过 |

---

## 五、Step 4：车辆检测与停留触发

遍历全部帧，检测画面中的车辆目标，并对异常候选车辆触发**高精度追踪模式**。

### 5.1 车辆检测与 ID 分配

使用帧差法、背景减除或目标检测模型，在每一帧中提取车辆目标。对连续帧中空间位置相近的目标分配唯一车辆 ID，建立跨帧追踪链。

记录每辆车的：
- 初始检测帧号
- 初始位置 (x, y, w, h)
- 外观特征（颜色、车型初步判断）

### 5.2 应急车道/路肩停留触发（核心触发器）

对每辆追踪中的车辆，持续监测其位置和速度。当满足以下**任一条件**时，立即触发高精度追踪模式：

- 车辆在应急车道或路肩区域停留时长 **>= 2 秒**
- 车辆在正常车道内连续 **5 秒** 以上无明显位移（跨帧位移 < 5 像素）
- 车辆行驶速度低于同车道其他车辆平均速度的 **20%**
- 车辆 ROI 区域在相邻帧间的模板匹配分数持续下降（提示异常运动模式）

> **重要规则**：一旦触发高精度追踪，对该车辆的判定**不允许直接标记为"违停"后跳过方向判定**。所有应急车道停留车辆必须完整执行 Step 5 和 Step 6 的方向分析流程。

---

## 六、Step 5：逐帧位移追踪（核心改进）

对全部车辆（尤其是高精度追踪模式下触发的车辆），执行**像素级精确**的逐帧位移追踪。

**本步骤只输出位移数据，不做任何方向判定。**

### 6.1 ROI 提取

在第 N 帧中，提取车辆所在的外接矩形区域，并向四周扩展 **20% padding** 作为 ROI。ROI 应包含车辆完整轮廓及周边局部路面纹理，用于模板匹配。

### 6.2 模板匹配定位

以第 N 帧的 ROI 作为模板，在第 N+1 帧的扩大搜索区域内进行归一化互相关匹配（`TM_CCOEFF_NORMED`）。记录最佳匹配位置及其匹配分数 score。

- 若 **score >= 0.7**：接受匹配结果，记录车辆中心坐标 (cx, cy)
- 若 **score < 0.7**：匹配失败，退而使用 ORB 特征点 + RANSAC 几何验证进行辅助定位
- 若辅助定位仍失败：标记该帧为"追踪丢失"，使用卡尔曼滤波预测位置，等待后续帧重新匹配

### 6.3 位移向量计算

对每辆车的连续帧中心坐标序列，计算逐帧位移：

- **dx** = cx_current - cx_previous
- **dy** = cy_current - cy_previous（Y方向位移，**核心指标**）
- **dt** = 当前帧时间戳 - 上一帧时间戳
- **瞬时速度 vy** = dy / dt（像素/秒）

### 6.4 轨迹数据记录

为每辆车建立完整的轨迹数据表，字段包括：

| 字段 | 说明 |
|------|------|
| frame_id | 帧号 |
| timestamp | 时间戳 |
| cx, cy | 中心坐标 |
| dx, dy | 位移 |
| vy | 瞬时速度 |
| score | 匹配分数 |
| status | 追踪状态（正常/丢失/恢复） |

> **关键原则**：车头朝向在本步骤中**仅作记录，不作为任何判定依据**。倒车行为的车辆车头朝向通常与正常车流一致，若将车头朝向用于异常判定，将直接遗漏所有倒车事件。

---

## 七、Step 6：逆行/倒车/违停判定

基于 Step 5 输出的精确位移轨迹，结合 Step 3 确定的主路正常行驶方向，执行最终的异常事件判定。

### 7.1 违停判定

对每辆车的完整轨迹计算累计位移：

- **正常车道**：若 |Σdy| < 阈值（建议 30 像素），且持续时间 >= 5 秒 → 判定为"违停"
- **应急车道**：若 |Σdy| < 阈值（建议 20 像素），且持续时间 >= 2 秒 → **进入 7.3 方向一致性判定，不允许直接标记违停**

### 7.2 方向一致性判定（核心判定）

计算车辆的整体位移趋势 Σdy，与其所在主路的正常行驶方向对比：

| 车辆整体位移趋势 | 主路正常方向 | 判定结果 |
|----------------|------------|---------|
| Σdy > +阈值 | Y增加（dy>0） | 正常行驶 |
| Σdy > +阈值 | Y减小（dy<0） | **逆行事件** |
| Σdy < -阈值 | Y增加（dy>0） | **逆行事件** |
| Σdy < -阈值 | Y减小（dy<0） | 正常行驶 |
| \|Σdy\| < 阈值 | 任意 | 违停（需进一步区分） |

### 7.3 逆行 vs 倒车细分

对 7.2 中判定为"逆行事件"的车辆，根据车头朝向进一步细分事件类型：

- **逆行**：位移方向与主路方向相反，**且车头朝向也与主路方向相反**（典型对向车道闯入）
- **倒车**：位移方向与主路方向相反，**但车头朝向与主路方向一致**（典型倒车/溜车行为）

车头朝向的判定方法：观察车辆前挡风玻璃、大灯、车牌位置等特征相对于行驶方向的前后关系。

> **车头朝向仅用于细分标签，不参与是否异常的核心判定。**

> **教训案例**：应急车道白色小车事件中，车辆车头朝南（与主路1正常方向一致），但实际位移方向为 Y 减小（向北）。原流程因仅粗略观察且未做精采样，将其误判为"违停"。正确判定应为**"应急车道倒车"**。

## 八、Step 7：多层验证机制

为防止单步骤误差导致误判，设计**四层交叉验证机制**。任何一层验证不通过，均需回溯修正。

### 8.1 验证 1：主路方向自洽验证

检查所有主路的行驶方向是否满足对向互补关系。

- **通过标准**：多主路场景下，任意两主路方向严格相反
- **未通过处理**：返回 Step 3 重新采样

### 8.2 验证 2：精采样重判验证（关键验证）

对 Step 4 中触发高精度追踪的所有车辆，分别用粗采样（1 FPS）和精采样（4 FPS）数据独立执行 Step 5 和 Step 6。对比两次判定结果：

| 粗采样结果 | 精采样结果 | 处理方案 |
|-----------|-----------|---------|
| 违停 | 违停 | 一致，维持违停判定 |
| 违停 | 倒车/逆行 | **以精采样为准，修正判定**（最常见的修正场景） |
| 正常 | 异常 | 以精采样为准，触发异常告警 |

### 8.3 验证 3：位移-方向交叉验证

对判定为"逆行"或"倒车"的车辆，执行可视化交叉验证：

1. 提取该车辆在首帧和尾帧中的位置截图。
2. 人工确认（或自动测量）尾帧位置减去首帧位置的符号，是否与主路正常方向相反。
3. 若可视化验证与数值判定冲突，以可视化结果为准，回溯检查模板匹配参数。

### 8.4 验证 4：应急车道强制判定验证

对所有在应急车道或路肩区域停留过的车辆，强制执行以下检查：

- [ ] 不允许直接标记"违停"后跳过方向判定
- [ ] 必须输出该车辆的逐帧 dy 序列
- [ ] 必须将该车辆的 Σdy 与其所在主路方向进行对比
- [ ] 必须在报告中独立列出该车辆的验证结果

## 九、Step 8：格式化报告输出

将全部分析结果输出为结构化的交通事件判别报告。

### 9.1 视频信息

文件名、监控点位 ID、录制时间范围、视频时长、分辨率、帧率、粗采样帧数、精采样帧数。

### 9.2 道路结构分析

主路数量、各主路画面坐标范围、正常行驶方向、判定依据车辆数、互补验证结果。

### 9.3 车辆追踪汇总

表格形式列出全部追踪车辆的基本信息：

| 车辆 ID | 类型 | 颜色 | 所在主路/车道 | 追踪帧数 | 总位移 Σdy | 平均速度 | 追踪置信度 |
|---------|------|------|-------------|---------|-----------|---------|-----------|
| ... | ... | ... | ... | ... | ... | ... | ... |

### 9.4 异常事件判定

对每个异常事件，输出以下信息：

- **事件类型**：逆行 / 倒车 / 违停 / 应急车道占用
- **车辆 ID**
- **车辆类型/颜色**
- **所在主路/车道**
- **发生时段**
- **总位移 Σdy**（含方向箭头）
- **主路正常方向**
- **判定逻辑链**
- **置信度**：高 / 中 / 低
- **关键证据帧编号**

### 9.5 验证结果

四层验证的逐项结果：

| 验证层级 | 状态 | 备注 |
|---------|------|------|
| 主路方向自洽验证 | 通过/未通过 | |
| 精采样重判验证 | 通过/未通过 | |
| 位移-方向交叉验证 | 通过/未通过 | |
| 应急车道强制判定验证 | 通过/未通过 | |

### 9.6 处置建议

根据事件类型给出分级处置建议：

- **一级（逆行/倒车）** → 立即拦截
- **二级（应急车道违停）** → 巡逻车核查
- **三级（异常慢速）** → 监控跟踪

## 十、关键规则速查表

| 场景 | 判定标准 | 常见陷阱 |
|------|---------|---------|
| 违停 | 累计位移 < 阈值，持续 N 秒 | 不要只看单帧位置，要跨帧求和；应急车道需触发精采样 |
| 逆行 | 位移方向 ≠ 主路方向，且车头方向 ≠ 主路方向 | 车头朝向仅用于标签细分，不用于异常判定 |
| 倒车 | 位移方向 ≠ 主路方向，但车头方向 = 主路方向 | **最容易误判为违停的场景！必须做精采样** |
| 应急车道停车 | 停留 >= 2 秒即触发精追踪 | 不能直接标记违停后跳过方向判定 |
| 精采样触发 | 应急车道停留 / 粗采样位移 < 5 像素但持续多帧 | 1 FPS 粗采样会遗漏慢速倒车 |

## 附录：工作流执行检查清单

在开始执行前，确认以下检查项：

- [ ] 视频已成功拆帧（粗采样 + 精采样）
- [ ] 所有帧按规范命名并保存在本地可访问路径
- [ ] 主路数量已判定并记录坐标范围
- [ ] 每条主路的方向已通过互补验证
- [ ] 全部车辆已分配唯一 ID 并建立追踪链
- [ ] 应急车道停留车辆已触发 4 FPS 精采样
- [ ] 精采样数据已独立执行方向判定
- [ ] 四层验证已全部执行并记录结果
- [ ] 输出报告符合 Step 8 的格式规范
